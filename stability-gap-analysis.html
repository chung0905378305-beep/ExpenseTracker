二、第一性原理：缺失功能分析

从"记账 App 会员管理后台"的完整业务闭环出发，审视 运营 → 增长 → 风控 → 维护 四个维度。 已完成度约 70%，以下是按优先级排列的缺失功能。

核心 CRUD
90%
安全与风控
85%
数据与分析
45%
运维与质量
20%

P0 必须补齐 上线前必须解决

缺失功能为什么必须影响
审计日志查看页面 AuditLog 表已在后端完整记录，但前端无查看入口。管理员无法追溯"谁在什么时候做了什么"，出问题只能直接查数据库。 P0
自动化测试 0 个单元测试、0 个集成测试。本次测试就发现 3 个致命 Bug，每次改代码都靠人工验证不可持续。 P0
数据库备份 SQLite 单文件数据库，无备份策略。会员数据、支付记录、激活码全在一个文件里，硬盘故障 = 全部丢失。 P0

P1 高优先级 影响运营效率和用户体验

缺失功能为什么重要影响
会员 / 支付数据导出 只有激活码导出 CSV。运营做周报/月报时无法导出会员列表、支付明细，只能截图或手动整理。 P1
会员标签 / 分层 无法按来源渠道、活跃度、付费金额给会员打标签，精细化运营无从下手。Model 里连个 tags 字段都没有。 P1
批量操作 不能批量封禁、批量修改套餐、批量生成激活码（已有）、批量导出。超过 10 个会员的操作只能一个个点。 P1
通知/消息系统 会员到期前无提醒、被封禁无通知。iOS 推送通知（APNs）和 App 内消息全是空白。 P1
Apple IAP 完整对接 Webhook 目前是骨架（仅打印日志）。真正的 Server Notifications v2 需要 JWS 签名验证、根据 SUBSCRIBED/DID_RENEW/EXPIRED/REFUND 事件自动更新会员状态。这是付费 App 的核心收入闭环。 P1
管理员 2FA 超级管理员只有一个密码保护。6 层安全防御防外部 AI 攻击，但管理员密码被钓鱼 = 全部沦陷。至少应加 TOTP。 P1
Swagger/OpenAPI 文档 40+ 个 API 端点无文档。iOS 开发者对接时需要逐行看后端代码，每次新增端点靠口头传达。Swagger 可自动生成 + 在线调试。 P1

P2 锦上添花 提升长期竞争力

缺失功能价值影响
收入预测与财务分析 MRR 已有，但缺少 ARR、LTV、CAC 回收期、MRR 增长率。无法做财务预测和定价策略调整。 P2
会员留存漏斗 不知道试用→付费转化率，不知道月度留存率。iOS App 订阅的本质是留存生意，没有这些数据就是盲飞。 P2
Docker 一键部署 目前需要手动 npx 启动、手动配置环境变量。没有 Dockerfile、docker-compose，换机器部署成本高。 P2
进程守护（PM2） Node 进程 crash 后不会自动重启。生产环境需要 PM2/systemd 守护。本次测试就 crash 了 3 次。 P2
结构化日志 & 监控 目前是 console.log，无日志级别、无日志文件轮转、无请求耗时统计。排查问题靠读源码。 P2
API Key 轮换 APP_API_KEY 是固定值，无轮换机制。如果泄露，只能在 .env 里手动改，然后重新部署。 P2
CI/CD 自动化 无 GitHub Actions 或任何 CI。每次部署是手工 scp/rsync。类型检查、测试、构建、部署全手动。 P2
三、架构层面缺失
层面现状差距
数据模型 8 个 Model，覆盖会员/套餐/订阅/支付/审计/权限/激活码/管理员 缺：会员标签 tags、通知推送 token、Apple transaction 原文存储、数据备份表
API 设计 RESTful，统一 res 格式，JWT + App Key 双鉴权 缺：分页 Link Header、API 版本废弃策略、批量操作端点、OpenAPI 文档
前端架构 Vue 3 + Pinia + Router，9 个页面，4 个组件 缺：错误边界组件、全局 Loading/Error 状态、表单统一校验、E2E 测试
安全体系 Helmet + CORS + RateLimit + JWT(jti/黑名单) + PoW + 输入消毒 + 审计 缺：2FA、IP 白名单、API Key 轮换、数据静态加密、请求签名
部署运维 手动 npx 启动，SQLite 本地文件 缺：Docker、PM2、日志系统、监控告警、备份策略、CI/CD
测试体系 本次稳定性测试脚本（Python） 缺：Jest 单元测试、Supertest 集成测试、Playwright E2E 测试
